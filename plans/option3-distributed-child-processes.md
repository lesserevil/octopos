# Option 3 Plan: Distributed Child Processes

## Purpose

This plan describes how to implement an OctopOS mode where child processes
created inside an `octopos exec` may run on other cluster nodes while the parent
process still sees usable Linux process behavior.

This is not a full transparent distributed kernel. The target is a practical
subset:

- Transparent remoting for common `fork -> exec` subprocesses.
- Local shadow PIDs so parents can `waitpid`, receive child exit status, and use
  common shell job control.
- Remote worker processes that run inside the same SSI rootfs on another node.
- Proxying for stdio, simple pipes, common signals, terminal resize, cwd, env,
  uid/gid, rlimits, and selected process metadata.
- Clear fallback to local execution when the child depends on IPC or kernel
  objects that cannot be represented safely across nodes.

Option 3 should be built in stages. Each stage must leave the system useful and
operational even if later stages are never completed.

## Remaining Implementation Backlog

The remaining work should be implemented in this order. Each phase should land
behind explicit opt-in behavior and include local tests plus at least one
bounded live validation on the `shedwards-octo*` cluster.

The IPC compatibility strategy is refined in `plans/ipc-compatibility-plan.md`.
That plan is authoritative for which IPC classes are implemented today, which
remain future work, which force local, and which are permanent non-goals.

### Phase 1: Finish Secure Child Control

Goal: make child-placement requests safe enough to use from transparent
interception code.

Tasks:

1. Add a dedicated child-spawn API instead of overloading generic `ExecStream`.
   - Define request/response structs or protobuf messages for child placement.
   - Include parent session ID, parent job ID, shadow PID, parent PID, resource
     hints, command, cwd, env, and FD plan summary.
   - Keep existing `octopos-remote-child` as a compatibility client on top of
     the new API.

   - Extract UID/GID/PID from the Unix socket.
   - Reject socket calls where peer credentials cannot be verified.
   - Keep internal node-to-node forwarding separate from local socket calls.
3. Harden token handling.
   - Attach token lifetime to the parent job.
   - Reject tokens after parent job reaches terminal state.
   - Rotate token on parent exec restart.
   - Ensure tokens never appear in job status, process listings, logs, or
     diagnostics.
4. Add authorization checks.
   - Caller session must match parent job session.
   - Caller may spawn only under its parent job/session.
   - Resource requests must be bounded by policy.
   - GPU requests must always pass through scheduler allocation.
5. Add audit records.
   - Parent PID, shadow PID, UID/GID, parent job, remote job, selected node,

6. Tests:
   - Valid token accepted.
   - Missing/wrong/expired token rejected.
   - Wrong session rejected.
   - Peer credential extraction tested where supported.
   - Remote forwarding cannot bypass local authorization accidentally.

Acceptance criteria:

- An exec process can spawn an explicit remote child through the local socket.
- A process outside an authorized exec cannot spawn a remote child through the
  socket.
- Internal node-to-node forwarding remains functional.
- All auth failures produce clear, non-secret diagnostics.

## Non-Goals

- Do not attempt arbitrary memory-state migration for a child that continues
  executing after `fork` without an immediate `exec`.
- Do not attempt transparent cross-node shared memory, futex, ptrace, pidfd, or
  Unix file descriptor passing in the first production version.
- Do not replace the Linux scheduler or modify the host kernel in the first
  version.
- Do not claim transparent static-binary or direct-syscall coverage until a
  future ptrace/seccomp hybrid is designed and validated. Production v1
  transparent remoting is the opt-in LD_PRELOAD runtime plus explicit
  `octopos-remote-child`.

## Definitions

- **Parent process**: The process running inside an OctopOS exec that creates a
  child.
- **Local shadow process**: A real local child process on the parent's node.
  This is the PID the parent can wait on and signal.
- **Remote worker process**: The real payload process running on a target
  OctopOS node.
- **Process coordinator**: The octoposd-side subsystem that tracks shadow to
  worker relationships and owns lifecycle decisions.
- **Remote-child runtime**: The user-space component loaded or invoked inside an
  exec namespace to decide whether a child should run locally or remotely.
- **FD proxy**: A byte-stream bridge between local file descriptors and remote
  file descriptors.
- **Placement policy**: Scheduler logic deciding whether a child is eligible for
  remote placement and which node should run it.

## High-Level Architecture

The first production architecture should use the local shadow process model.

1. The parent creates a child in the local kernel.
2. The child becomes an OctopOS shadow process.
3. The shadow contacts local octoposd over a trusted local control channel.
4. octoposd schedules and starts a remote worker via the existing ExecStream
   infrastructure or a new lower-level remote-worker RPC.
5. The shadow proxies stdin, stdout, stderr, signals, terminal resize, and exit
   status.
6. The parent waits on the local shadow PID.
7. OctopOS metadata records the mapping from local shadow PID to remote worker.

This gives the parent a normal child PID without pretending the local kernel
contains a remote task.

| IPC or process interaction | Support level | Notes |
| --- | --- | --- |
| stdin/stdout/stderr | Required | Proxy byte streams between shadow and remote worker. |
| Shell pipelines | Required for common cases | Proxy anonymous pipe endpoints when byte-stream only. |
| Exit status and `waitpid` | Required | Parent waits on local shadow. Shadow exits with remote status. |
| `SIGCHLD` | Required | Provided naturally by local shadow exit. |
| `SIGINT`, `SIGTERM`, `SIGHUP`, `SIGQUIT` | Required | Forward from shadow/process group to remote worker. |
| `SIGSTOP`, `SIGCONT` | Phase 2 | Needed for shell job control. |
| Terminal resize | Required for PTY sessions | Forward window size to remote PTY. |
| Process groups and sessions | Phase 2 | Required for interactive shells and job control. |
| Regular files on SSI rootfs | Required | Rely on shared cluster FS. |
| File locks | Phase 2/3 | Depends on cluster FS lock semantics. Must test carefully. |
| TCP and UDP using routable addresses | Required | Normal network paths work. |
| Anonymous pipes | Required for simple streams | Force local for non-byte-stream assumptions. |
| Named FIFOs | Supported for simple streams | Blocking `O_RDONLY`/`O_WRONLY` SSI-root opens are brokered. |
| Unix pathname sockets | Supported for streams | Pathname `AF_UNIX`/`SOCK_STREAM` sockets under the SSI root are brokered. |
| Localhost TCP | Unsupported transparently | Remote worker has its own localhost. Provide diagnostics. |
| Unix abstract sockets | Unsupported initially | Kernel-local. Force local. |
| `SCM_RIGHTS` fd passing | Unsupported initially | Very hard. Force local. |
| POSIX/System V shared memory | Unsupported | Force local. |
| Shared-memory futexes | Unsupported | Force local. |
| POSIX/System V message queues and semaphores | Unsupported initially | Kernel-local. Force local. |
| `eventfd`, `signalfd`, `timerfd` | Unsupported initially | Local-only unless explicitly proxied later. |
| `pidfd` | Unsupported initially | Use shadow PID only. |
| `ptrace` | Unsupported | Force local. |
| `inotify` and `fanotify` | Partial later | Shared FS events are subtle. Force local initially. |
| Netlink | Local only | Force local for commands depending on netlink state. |
| `mmap(MAP_SHARED)` | Relaxed read-only file-backed support | Writable or shared-memory-backed mappings remain unsupported. |

## Compatibility Contract

The system must make eligibility explicit and inspectable.

If OctopOS remotes a child, it must be able to explain why. If it keeps a child
local, it must also be able to explain why.

Example diagnostic output:

```text
octopos: child 18432 remoted to shedwards-octo3 as worker 9981
octopos: child 18450 kept local: fd 5 is unix socket
octopos: child 18461 kept local: process called fork without exec
```

Diagnostics should be configurable:

- Off by default for normal use.
- `OCTOPOS_REMOTE_CHILD_DEBUG=1` for per-process stderr diagnostics.
- `octoposctl job status` should always expose placement decisions.

## Milestone 5: Interception Runtime

Goal: intercept common child creation paths and replace eligible children with
shadow processes.

Start with opt-in dynamic-binary interception. Do not make it default.

Candidate approaches:

1. **LD_PRELOAD runtime**
   - Intercept `posix_spawn`, `execve`, `fork`, `vfork`, and `system`.
   - Useful for shell/Python/Make workloads.
   - Not sufficient for static binaries or direct syscalls.

2. **ptrace supervisor**
   - More complete than LD_PRELOAD.
   - Higher overhead.
   - Complex signal interactions.

3. **seccomp user notification**
   - Useful audit mechanism for syscall visibility.
   - Not a remoting mechanism in production v1.

Recommended path:

- Treat the opt-in LD_PRELOAD runtime as production v1 for dynamic binaries.
- Keep seccomp user notification audit-only for static/direct syscall visibility.
- Defer ptrace/seccomp remoting until there is workload evidence that justifies
  the extra signal, wait, and security complexity.
- Keep the remote-child decision API independent of interception mechanism.

Tasks:

1. Add `cmd/octopos-child-supervisor` audit mode and the LD_PRELOAD runtime.
2. Intercept `execve` after `fork` for common shell workloads.
3. For eligible commands:
   - create local shadow process
   - launch remote worker
   - proxy streams
   - exit with remote status
4. For ineligible commands:
   - execute locally
   - record fallback reason
5. Add env controls:
   - `OCTOPOS_REMOTE_CHILDREN=off|safe|aggressive`
   - `OCTOPOS_REMOTE_CHILD_DEBUG=1`
   - `OCTOPOS_REMOTE_CHILD_FORCE_LOCAL=1`
6. Add command allow/deny patterns:
   - Allow common build tools and batch commands.
   - Deny service managers and tools known to need local kernel state.

Acceptance criteria:

- `bash -c 'hostname'` can remote the child in prototype mode.
- `make -j` can remote simple compile commands in prototype mode.
- Commands with unsupported fds remain local.
- Static binaries still run locally rather than failing.

Suggested junior developer tasks:

- Implement LD_PRELOAD wrappers.
- Add command policy parser.
- Add fake coordinator tests.
- Add shell examples.

## Milestone 6: Pipes and Shell Pipelines


## Milestone 11: Failure Semantics

Goal: make failures unsurprising.

Define behavior for each failure:

| Failure | Required behavior |
| --- | --- |
| Remote worker exits | Shadow exits with same code or signal. |
| Shadow receives SIGTERM | Forward to remote worker, then exit after worker exits. |
| Parent exits | Follow configured parent-death policy. Default kill remote child. |
| Local octoposd restarts | Mark shadow mappings recovering, reconcile against the worker node, or orphan after lease expiry. |
| Remote octoposd restarts | Kill or mark remote worker failed. |
| Remote node dies | Shadow exits with synthetic failure code and diagnostic. |
| Network partition | Timeout, then policy: kill, detach, or fail. |
| Proxy stream breaks | Close corresponding remote fd and finish lifecycle. |

Tasks:

2. Define signal mapping for remote signal death.
3. Add remote heartbeat for workers.
4. Add shadow-to-worker lease.
5. Add recovery and orphan cleanup on daemon startup.
6. Add user-visible diagnostics.

Acceptance criteria:

- Remote node failure does not leave scheduler resources allocated forever.
- Shadow exits when remote worker cannot be recovered.
- `job status` explains infrastructure failure.

## Key Design Risks

### Risk: Remote fork without exec

Arbitrary `fork` creates a child with copied memory, locks, and runtime state.
Remote execution cannot preserve that without checkpoint/restore or kernel
support.

Mitigation:

- Only remote at the exec boundary.
- Force local when a child runs meaningful code after fork before exec.
- Document this explicitly.

### Risk: Unsupported inherited file descriptors

Many Linux IPC objects are local to one kernel.

Mitigation:

- Classify fds before remoting.
- Default unknown fds to local.
- Emit clear diagnostics.

### Risk: Shell job control complexity

Process groups, terminal foreground state, stop/continue, and PTY behavior are
subtle.

Mitigation:

- Defer full job control until after non-interactive workloads work.
- Add focused manual validation scripts.
- Keep a force-local escape hatch.

### Risk: Daemon restarts interrupt children

Existing foreground exec streams can be coupled to octoposd lifetime.

Mitigation:

- Keep shadow processes outside octoposd cgroup where needed.
- Use leases and reconnect.
- Make kill/detach policy explicit.

### Risk: Security boundary confusion

Processes inside an exec should not be able to request arbitrary privileged
actions on the host.

Mitigation:

- Use local socket credentials.
- Use per-exec tokens.
- Validate session/job ownership.
- Sanitize env and resource requests.
- Audit every remote child request.

## Open Questions

1. Should remote children be opt-in per exec or per session?
2. What is the default policy when a remote node dies: fail shadow, detach, or
   retry elsewhere?
3. Should remote child placement ever happen for root-owned package install
   scripts?
4. How much `/proc` identity should point to the shadow versus the remote
   worker?
5. Should `localhost` be rewritten or explicitly documented as node-local?
6. Should process remoting be disabled by default when a GPU is allocated to the
   parent?
7. What exit code should represent OctopOS infrastructure failure?
8. Do we need persistent shadow records across octoposd restarts, or is
   in-memory state sufficient for v1?
